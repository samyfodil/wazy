# Optimization Scan

Baseline: wazero upstream `c0f3a4e` ("module: only release owned memory on module close").
Method: six subsystem scans (compiler backend, interpreter, runtime core, WASI/sysfs, API/cache
layer, hostcall mechanism). Findings marked **measured** were verified with pprof, `-gcflags=-m`
escape analysis, `-d=ssa/check_bce`, or in-repo benchmarks on this checkout.

## Top priorities (cross-cutting, ranked)

| # | Finding | Area | Impact |
|---|---------|------|--------|
| 1 | Reflect-wrapped host functions 14.5x slower than direct (1086 vs 75 ns/op, 6 allocs) — no fast-path adapters for common signatures | hostcall | high, **measured** |
| 2 | Interpreter: heap-allocated `*callFrame` per call (~1 alloc/call; 1.35M allocs/op in fib) | interpreter | high, **measured** |
| 3 | Interpreter dispatch loop too big for Go inliner — `popValue` alone 13.6% CPU as a real call | interpreter | high, **measured** |
| 4 | `fd_readdir`: one lstat(2) + FileInfo alloc per dirent (~11x slower than needed) | sysfs | high, **measured** |
| 5 | `try_table` enter: Go-exit + full stack clone + 1KB register copy on the non-throwing path | compiler | high |
| 6 | `ExportedFunction()` allocates a fresh callEngine + 10 KiB zeroed stack every lookup | API | high |

## 1. Hostcall mechanism

Measured baseline (amd64): direct `api.GoModuleFunc` round trip **74.9 ns/op, 0 allocs**;
the original reflection-based host-function registration **1086 ns/op, 6 allocs** (14.5x).

- **H1. Reflect path has no fast path** — historical: per call the reflection path did `make([]reflect.Value, pLen)`, `reflect.New(t).Elem()` per param, `newContextVal`/`newModuleVal`, and `fn.Call`. Impact: high, **measured**, certain.
  - **RESOLVED**: reflection was removed entirely. Host functions are now registered either through the explicit `WithGoModuleFunction`/`WithGoFunction` stack API or through the typed generic helpers `wazy.HostFunc0`-`HostFunc8` / `HostProc0`-`HostProc8` (`host_typed.go`), which derive the WebAssembly signature from Go's type system and decode/encode the stack directly. Both are zero-allocation and at parity with direct `GoModuleFunc`. The interim generated reflect-adapter matrix (`gofunc_fast.go`) was deleted along with the reflect registration path.
- **H2. itab hash lookup per host call** — **RESOLVED**: `buildHostModuleOpaque` now type-switches once and stores the asserted `api.GoModuleFunction`/`api.GoFunction` interface words; `hostModuleGoFuncFromOpaque[T]` reinterprets without `assertE2I`. Combined with H5+A7: gomodule CallWithStack 82→74 ns, now ~11.5% faster than upstream.
- **H3. Trampoline saves 23 regs one at a time (arm64)** — **RESOLVED**: save/restore now pairs adjacent same-type registers via new `stp`/`ldp` signed-offset encodings (`storeP128`/`loadP128` kinds); go-call trampoline 23→12 memory instrs each way, stack-grow 39→20. Validated by ARM ARM hand-computed encoding tests, independent disassembly, and full spectest v1/v2 + engine tests on arm64 under qemu. amd64 has no pair-store; left unchanged. (C9 prologue pairing can reuse the new encodings.)
- **H4. Listener path eagerly unwinds the whole stack** before `listener.Before` even if unused; per-call `FunctionDefinition` lookup. `call_engine.go:399-460`. Fix: lazy iterator, cache defs. Impact: medium (listener users only), certain.
- **H5. Non-inlinable closure + conditional defer wraps every host call** — **RESOLVED**: the four dispatch sites branch to named `callGo*WithSnapshotRecover` functions only when the snapshotter is enabled; common path calls `f.Call` directly (no closure, no defer).
- **H6. `ensureTermination`: loop-header check compiles to a full Go-exit round trip per loop iteration** — `frontend/lower.go` — **RESOLVED (amortized; native-flag disproven)**. The originally-proposed fix (native load+branch on a closed flag, no Go round-trip) was **empirically proven unsafe**: the per-iteration Go round-trip is the *only* scheduler/GC yield point in an otherwise non-preemptible compiled loop (comment at `call_engine.go` `ExitCodeCheckModuleExitCode`), so a pure native flag would let a tight `WithCloseOnContextDone` loop block Go's GC stop-the-world for its entire duration — measured: `runtime.GC()` blocked **6.35 s** during a spin loop with no yield point, vs 539 µs once it ended. This is why upstream shipped [#2482](https://github.com/tetratelabs/wazero/pull/2482) (amortize the yield) rather than a flag. Ported that design and extended it: loop headers bump an `interruptCounter` in the execution context (new field, offset 1248) and only do the Go round-trip when `(counter & (interval-1)) == 0`. **wazy defaults the interval to a non-zero 64** (fast out of the box; upstream defaults 0/unchanged), overridable **per compile** via `wazy.WithInterruptCheckInterval(ctx, n)` (0 or power of two; validated at `CompileModule`). The interval is folded into the module ID (`AssignModuleID`), so distinct intervals are distinct cached variants — **re-lowering a module is a fresh `CompileModule` under a context carrying a new interval** (the seam for a future runtime relower). Verified on both ISAs (amd64 + arm64/qemu): a tight infinite loop under `WithCloseOnContextDone` now keeps GC unblocked (`runtime.GC()` < 500 ms) *and* still terminates on context deadline; counter-path loop results correct for intervals {0,1,8,64}. NOT done: the per-entry `CloseModuleOnCanceledOrTimeout` watcher goroutine remains (per-entry, not per-iteration — minor). **Measured** (quiet i9-12900HK, benchstat n=10, `benchmarks/vs-wazero` `BenchmarkCloseOnContextDone`, a 100k-iteration tight loop): under `WithCloseOnContextDone` wazy **160µs vs wazero@main 2074µs, -92.3% (~12.9x faster)**; with the feature off wazy is on par (+2.75%, no regression). Real caseWasm fibonacci (loop-based): wazy **6.56ms vs wazero 76.0ms, -91.4% (~11.6x)**. The overhead scales to loop-body cost: on a **realistic loop calling a host function each iteration**, wazy's on-vs-off overhead is **+5.3%** (wazero's is +75.5%), and wazy is 2.0x faster than wazero close-on. Only near-empty compute kernels (spin, fib-inner) pay the 1.7-2.4x worst case — measured via the `WithInterruptCheckInterval` sweep (interval 0→4096: 26.9x→1.74x-of-floor, the residual being the per-iteration counter bookkeeping that no interval amortizes). Smart mitigations considered and rejected: yield-point elision (skip the check on loops that already yield) needs dominance analysis + moving cancellation observation off the loop check onto host-call return (`FailIfClosed` fires only at the check today), and would only help the very kernels it cannot make free; a register-resident (loop-carried) counter needs threading a value through every back-edge for a ~1/3 cut. Neither beats the fundamental cost of interrupting a pure-compute tight loop, so the amortized counter stays and the feature remains opt-in (default-on would invert the compute-kernel benchmarks vs wazero). Impact: ~64x fewer Go round-trips in `WithCloseOnContextDone` tight loops.
- **H7. Trampoline stack-bounds check** — **RESOLVED**: every function prologue now guarantees a `StackBoundsCheckMarginBytes` (512) slack margin below its (fixed-size) frame, so the go-call trampoline skips its own bounds check whenever the host signature's Go arg/result slice fits the margin (the common case; huge-arity signatures keep the check). Check-only change (frame allocation / `FrameSize()` untouched, so EH SP/FP recovery is unaffected — cross-checked); margin is shared headroom (~512B/stack, not per-frame). Measured on a quiet machine, tightly interleaved + core-pinned: gomodule host-call CallWithStack -1.01% (p=0.002); typed -0.2% (n.s.). Invariant (fixed frames → sp constant in-frame → margin survives to the call site) guarded by `TestNativeDeepRecursionWithHostCalls` + the StackGuardCheckEnabled canary. Both ISAs, incl. arm64 spectest/EH/deep-recursion under qemu.
  - **H7-followup (RESOLVED): margin was in the wrong place — it inflated the prologue probe immediate.** Adding MARGIN into `requiredStackSize()` pushed the probe's `sub/add $imm,%rsp` past the amd64 imm8→imm32 boundary (128) for otherwise-small frames, adding +6 bytes to the hot prologue. On `Case3/fibonacci` (pure recursion, ~2.7M direct calls, requiredStackSize 96→608) that shifted the tight loop across a fetch/uop-cache window: **IPC 4.19→3.15, ~23% slower than fork-point wazero** (byte-identical body; the whole delta was the two probe immediates). Found by three-way disassembly diff vs pinned wazero; proven by a margin sweep (binary threshold at 128) + `perf stat`. **Fix:** reserve MARGIN by biasing the *stored* `stackBottomPtr` check-limit up by MARGIN (all writes routed through `stackCheckLimitPtr`, `call_engine.go`) and drop it from `requiredStackSize()` (both ISAs). The check inequality `frameBase − realBottom ≥ maxCalls+frameSize+MARGIN+32` is provably unchanged — only which side carries MARGIN moved — so H7's trampoline-elision and the C24 leaf-elision invariants are untouched; the hot-path immediate returns to imm8 (fib 608→96). **Restores parity with fork-point wazero** on `Case3/fibonacci`: min-of-8 interleaved on a verified-quiet i9-12900HK, wazy 2.849ms vs wazero 2.798ms (was 1.23× → now 1.02×, i.e. the 23% regression eliminated). fib's emitted code is now identical to wazero's (body was already byte-identical; both prologues now `sub $96`), so the residual ~2% is mmap-placement noise. (Still ~16% behind wasmtime/Cranelift on this pure-compute recursion — a separate raw-codegen gap, not this regression.) Adversarially audited (predicate equivalence both ISAs, all 5 limit-write sites, grow-path sizing, pointer-in-range, leaf/trampoline invariants); amd64 `go test ./...` green + arm64 under qemu (deep-recursion/stack-overflow/grow/snapshot/`memory_grow_in_recursive_call`, 354 pass). Field kept named `stackBottomPtr` (asm-offset symmetry) but re-documented as the biased limit.
- **H8. Interpreter: per-hostcall interface type switch** — `interpreter.go:871-877`. Normalize to `GoModuleFunction` at compile. Impact: low, certain.
- Verified good: exit mechanism is store+ret (no thread/goroutine switch); one copy each way for params; listener-ness baked into exit codes (zero cost when unlistened); no `context.WithValue` per hostcall.

## 2. Interpreter (`internal/engine/interpreter`)

pprof on fib: `callNativeFunc` 59% flat, `popValue` 13.6%, `pushValue` 6.8%, malloc ~10%.

- **I1. `*callFrame` heap alloc per call** — **RESOLVED**: `frames []callFrame` value slice; frame pointer re-taken after every nested call. fib_for_30 allocs/op: 1,346,273 → 2.
- **I2. Dispatch func is 'big' (cost 56600) → inliner cap drops to 20** — **RESOLVED**: SIMD/atomics/bulk-memory cases (79 ops) split into `callNativeFuncRare`; cost 56600 → ~11350, `popValue`/`pushValue`/`drop`/`popMemoryOffset` (289 sites) now inline into the dispatch loop.
- **I3. `frame.pc` lives on heap** — **RESOLVED**: local `pc` register mirror with documented sync policy (traps/calls/throws sync, ALU/branch path store-free); loop guard indexes `len(body)` directly so the per-instruction bounds check is eliminated. Trap stack traces verified byte-identical. I1+I2+I3 combined: fib_for_20 ~1.8 ms → ~1.05-1.2 ms, fib_for_30 ~184-211 ms → ~125-147 ms (~30% faster), allocs flat 2/op.
- **I4. `ctx.Value(EnableSnapshotterKey{})` per call instruction** — `interpreter.go:252`. Fix: compute once per activation. Impact: med-high, certain. — **ALREADY RESOLVED (verified) by the `SnapshotterEnabled` latch (H5/A7).** The per-call-instruction site (`callWithUnwind`) and the per-activation site (`call`) both gate the `ctx.Value` chain walk behind `expctxkeys.SnapshotterEnabled.Load()` — a plain atomic-bool load that is false unless some context in the process has *ever* enabled the snapshotter. So in the common case there is no ctx walk per call, only a cheap load; the expensive chain traversal the item flagged is already elided. No per-*opcode* (dispatch-loop) snapshotter check exists. Nothing further to do without also hoisting the atomic load itself (negligible).
- **I5. Label ops compiled in and executed as no-ops**; `br`-to-next pairs emitted (`compiler.go:506,739,673`). Fix: strip labels + remap PCs in `lowerIR`; peephole fallthrough brs. Impact: medium, certain. — **RESOLVED (the load-bearing 90% — a cheap dispatch case; full strip deferred as marginal + high-risk).** First nailed down the mysterious execution path: `operationKindLabel` ops **are** in the runtime `body` and pc **does** land on them (fall-through or as a branch target — a loop header runs its label every iteration), confirmed by a counter probe. They were no-ops only by accident of control flow: `callNativeFuncRare` has **no `default` arm** (its final grouped case for the generic unspecialized Kinds panics, but any *unlisted* kind — Label included — matches nothing and silently returns), so each label was executing as: main-dispatch `default` → **a full non-inlined call to the ~2,900-line `callNativeFuncRare`** → no match → return → `pc++`. The fix diverts them: an explicit `case operationKindLabel: pc++` in the hot `callNativeFunc` switch, so a label costs one increment instead of a function call. Zero index remapping, zero correctness surface. **Measured** (interpreter, `BenchmarkCase3Interp` wazy before/after, interleaved sample-by-sample + core-pinned so the concurrent load cancels — the reliable-under-load method): **−9.03% geomean** — fibonacci **−14.4%** (recursion runs each call body's block/if labels ×2.7M calls), reverse_array −10.7%, string_manipulation −9.0%, random_mat_mul −6.6%, base64 −4.2%; all p=0.000, n=12. Full `go test ./...` green (labels are still present and still no-ops, just cheaply). The *full* strip (remove labels from `body` and remap every branch target + the index-aligned `IROperationSourceOffsetsInWasmBinary` + exception-table PCs) would additionally save the per-label `pc++`/dispatch step and shrink `body`, but that's marginal on top of −9% for a broad, correctness-critical remap — deferred. "Peephole fallthrough brs" (a `br` whose target is the next instruction) is a separate small follow-up, also deferred.
- **I6. Two-level dispatch** — **RESOLVED**: `lowerIR` specialization pass rewrites generic ALU/compare/load/store kinds into 72 monomorphic kinds; inner type switches deleted from dispatch. Hot/rare split rebalanced with dynamic-frequency data (no relocated op >0.26% in real workloads).
- **I7. `LabelCallers` map written per branch, never read** — `compiler.go:258` + ~20 sites; cleared with per-key deletes. Fix: delete it. Impact: medium (compile time), certain. — **RESOLVED.** `compilationResult.LabelCallers` was populated at 17 branch-lowering sites and cleared per function via a nested `frameID × labelKind` delete loop in `Next()`, but never read anywhere (confirmed: only writes/deletes, and the interpreter has no on-disk cache that could serialize it). Deleted the field, all 17 writes, the per-function clear loop, and the init; updated the `compiler_test.go` expected `compilationResult`s. **Measured** (interpreter compile of the 90-function case.wasm, `-benchmem`): **566 → 555 allocs/op (−11), 1161.7 → 1157.0 KiB/op (−4.7 KiB)** — the map was allocated once and reused (clear-not-realloc), so the win is its initial alloc + growth churn + the eliminated per-function clear loop; deterministic (alloc counts stable across runs). Interpreter + spectest v1/v2 + integration engine green. Pure dead-code removal, zero runtime-behavior change.
- **I8. `unionOperation` is 56 B**; `Us []uint64` header dead for ~95% of ops. Fix: side pool → 32 B ops. Impact: medium, likely. — **RESOLVED.** The 24-byte `Us []uint64` header (used by only 4 op kinds — BrTable targets/ranges, V128Shuffle lanes, TailCallReturnCallIndirect / ReturnCallRef `[dropDepth,label]`) is removed from `unionOperation` and moved to a per-function side table (`compiledFunction.sideTable [][]uint64`); those ops store their side-table index in `U3` (which all four otherwise leave free). Reordering wasn't needed: dropping the slice header alone takes the struct **56 → 32 B**, so the hot `body []unionOperation` dispatch array is 43% denser (two ops per 64-B cache line instead of ~one). The side table is accumulated per function during compilation (`compilationResult.SideTable`, `addSideTable`, reset in `Next()`), cloned into the `compiledFunction` in `lowerIR` (outer clone, inner slices shared exactly as `body` shares operand backing — label resolution still mutates entries in place), and read at the ~5 dispatch/resolution sites via `frame.f.parent.sideTable[op.U3]`. **Measured** (deterministic, load-independent — compile of the 90-function case.wasm): total op body **719 KiB → 411 KiB (−42.9 %, −308 KiB)** for 12,841 ops, with only **19 side-table entries** across all 90 functions (confirming `Us` is dead for ~99.9 % of ops here). This is real resident-memory savings for long-lived cached compiled functions; the runtime d-cache-density upside (denser hot dispatch array) is a bonus that wasn't separately timed — interpreter wall-clock needs a quiet box and the machine was loaded — but the memory reduction alone justifies the change. **Correctness**: full `go test ./...` green, incl. all spectest suites v1/v2 (br_table, `simd` shuffle) + tail-call + typed-function-references + exception-handling + threads and the `integration_test/engine` suite, on both engines. Debug `String()` for BrTable now prints the side-table index (the value-receiver can't reach the table); one unit assertion (`TestCompile_Vec` shuffle) shifted from checking inline lanes to the pooled form, lane values still covered end-to-end by the simd spectests.
- **I9. Per-call chain `callWithUnwind`→`callFunction`→3-way dispatch**, all non-inlinable, static per callee. Fix: callee-kind tag, direct dispatch. Impact: medium, likely. — **ASSESSED, NOT PROCEEDING (measured neutral).** Confirmed all four (`callWithUnwind`, `callFunction`, `callNativeFunc`, `callGoFuncWithStack`) are non-inlinable, so the common wasm call really is `callWithUnwind → callFunction → callNativeFunc` = two call layers. Prototyped removing the middle one on the hot path: in `callWithUnwind`'s fast path, dispatch straight to `callNativeFunc` when the callee has no host func / listener (the common case), keeping `callFunction` only for the rare host/listener callees. **Measured on fibonacci — the most call-heavy export (~2.7M calls/op), i.e. the best case for this — interleaved + core-pinned, n=16: 157.4 → 155.9 ms, −0.95%, p=0.061 (not significant).** Removing one non-inlinable call layer saves essentially nothing: Go's call overhead is tiny next to `callNativeFunc`'s body, and this is the *most* call-bound workload — loop-heavy exports (fewer calls) would show even less. A callee-kind tag would only shave two field derefs on top, strictly smaller. Not worth adding a duplicated hostFn/listener branch to the hot path for a sub-1%, borderline-noise delta. Reverted (same verdict shape as C6).
- **I10. Memory ops: double bounds check + 2 non-inlined calls per load** — **RESOLVED**: specialized load/store cases use one merged bounds check (proven equivalent incl. the 4GiB boundary) + direct `binary.LittleEndian` access. I6+I10+I11 combined: ~3.9% geomean on interpreter benchmarks (interleaved benchstat; smaller than scanned estimate because I1-I3 already inlined the helpers these paths used).
- **I11. Global get/set chases `g.Type.ValType` to detect V128** — **RESOLVED**: V128-ness precomputed into `op.B3` at lowering.
- **I12. `BrIf` calls non-inlined `drop` even for the no-drop common case** — check the -1 sentinel inline. Impact: low-med, certain. — **ALREADY RESOLVED (verified).** The premise no longer holds: the I2/I3 dispatch-loop split made `(*callEngine).drop` inlinable, and `-gcflags=-m` confirms both `drop` and `inclusiveRangeFromU64` are **inlined at the `operationKindBrIf` site** (interpreter.go:1181). So the taken-branch path has no call — the `Start == -1` early-return is already inline in the dispatch loop, exactly what the item asked for. An explicit outer sentinel guard would only duplicate the inlined check.
- I13/I14 (compile-time): `lowerIR` zero-extend temp allocs; `Next()` per-key map clear. Low.

## 3. Compiler backend (`internal/engine/native`)

- **C1. `try_table` enter/leave: Go-exit + `cloneStack` (≥10KB) + 1KB `savedRegisters` copy per dynamic entry** — **RESOLVED (P0+P1)**: per-function PC-range exception side table (`nativeapi/eh_table.go`) + native stack walk at throw time replaces the stack clone; enter no longer clones (still keeps a 1KB register snapshot as the P1 safety net — P2 removes it). Throw resumes the catching frame at its enter-continuation with the matched clause index, so control transfer is identical in shape to the fast-path exit on both ISAs (unified amd64+arm64; arm64 verified under qemu). Bench: EH fast path -91%, throw path -99.7%, EH allocations -99.5%. **P2a**: the per-enter heap save-area allocation is now pooled (one reusable buffer per nesting depth, stack-disciplined), removing the last per-enter heap alloc — fast path a further -23%, allocations flat 2/call regardless of enter count. The residual 1KB register snapshot per enter is NOT yet removed: doing so needs frame-slot memory homes not just for locals/operand-floor (§4.1) but also for `moduleCtxPtrValue`, which handler blocks reload and which lives in a callee-saved register across the enter call — a real Emscripten C++ module crash surfaced this gap the design missed. P2b (register-snapshot removal) was attempted and proven INFEASIBLE frontend-only: `execCtxPtrValue` is whole-function-live and every memory home is addressed through it (circular), and native gives the frontend no frame-slot facility. A poison experiment confirmed the callee-saved restore is load-bearing even for the simplest throwing module (empty floor, no memory/globals). Truly zero-cost enter needs a backend abnormal-edge ABI (P3, out of the design's P2 scope). **P3 outcome**: P3.0 (fixed execCtx/moduleCtx frame slots, gated on EH functions) shipped; the P3.2 regalloc abnormal-edge was validated feasible; but P3.1 (drop the snapshot) is proven BLOCKED — a real C++ workload (`test_catch_specific`) showed a *further* callee-saved category survives beyond execCtx/moduleCtx/locals/floor, so no frontend-tractable memory-homing subset is safe (third incorrect enumeration of a landing pad's needs). Dropping the snapshot needs a general regalloc abnormal-edge liveness model — disproportionate. **C1 is effectively complete at P3.0**; the residual per-enter 1KB snapshot is the practical floor. Net C1: throw path -99.7%, fast path ~-93% cumulative, per-enter heap alloc eliminated. Register snapshot stays; durable adversarial EH tests added. See docs/design/eh-p3-backend-abi.md OUTCOME.
- **C2. amd64: 2 defensive reg copies per binary ALU/shift/xmm op** — `amd64/machine.go` `lowerAluRmiROp`/`lowerShiftR`/`lowerXmmRmR` (~1726/1753/1789). **INFEASIBLE (proven — coalescing already realizes the win).** The lowerings emit `copy1: mov rn->tmp; alu tmp OP= rm (defKindNone); copy2: mov tmp->rd`. copy2 is load-bearing: `aluRmiR`/`shiftR`/`xmmRmR` are `defKindNone` (instr.go `defKinds`), so the ALU op sets no `vs.defInstr`; only the post-op copy is a single-Def instr and becomes rd's spill anchor (`regalloc.go:894`). `scheduleSpill` inserts rd's store via `StoreRegisterAfter(rd, defInstr)` (`regalloc.go:1171`). The naive "compute into rd directly" (`copyTo(rn,rd); alu rd OP= rm`) MISCOMPILES: it makes the *pre*-alu copy rd's only Def, so rd's spill store is scheduled BEFORE the alu computes → stale value (empirically: `tail_call_return_call` returned 0). The narrow variant — gated on `xDef.RefCount==1` (source dies here): `alu rn OP= rm; copyTo(rn,rd)` — keeps the post-op copy as anchor and IS provably correct (spectest v1/v2 incl. tail-calls pass; the defKindNone value-model discrepancy is benign exactly as it already is for `tmp`: rn is reserved for x through its `lastUse` = the post-op mov since `lastUse` is computed from emitted machine uses `regalloc.go:664-671`, `aluRmiR` reports op2 as a Use, and x is single-use so nothing else reads it). BUT it delivers ZERO benefit: for single-use x the original's copy1 already coalesces away in the regalloc def phase (`regalloc.go:862-867`: a copy whose source reg is free — x killed at copy1 — reuses it, becoming a self-move dropped at finalize). Measured on all 90 functions of the bench workload (`bench/testdata/case.wasm`, incl. fibonacci/mat_mul/base64): variant vs HEAD finalized codegen has an **identical opcode histogram**, identical instruction count (31661), identical spill count (3728) — only a handful of register-name permutations. So runtime is provably identical; interleaved `BenchmarkCompilation` showed no compile win (noise, trended +6% at load ~12). The real (non-coalesced) win would need a defKindNone spill-anchor model in regalloc — disproportionate to a runtime benefit coalescing already fully captures. Not worth the added lowering complexity + correctness surface (the shift lowering additionally introduces a live-range-across-fixed-rcx hazard the fresh-temp path avoids). Verdict: leave as-is.
- **C3. arm64: `Ishl` never folded into scaled addressing** (in-code TODO, `lower_mem.go:271`) — every scaled array access = lsl+add+ldr instead of `ldr [base, idx, lsl #n]`. Impact: high, certain. — **IMPLEMENTED.** `collectAddends` now folds a single-use `index << log2(size)` (when the shift equals the transfer size's scale) into the address mode's scaled index, emitting the `addressModeKindRegScaled` / `addressModeKindRegScaledExtended` modes the encoder already supported but selection never produced. Three forms: 64-bit index → `[base, x, lsl #n]`; 32-bit index behind a single-use U/SExtend → `[base, w, {U,S}XTW #n]` (the extend is consumed too); shift-mismatch / multi-use / no-base falls back to the prior lsl+add path. Collapses the common `arr[i]` load/store from lsl+add+ldr to a single scaled ldr/str. Correctness verified on arm64 under qemu: full spectest v1+v2, native engine suite, backend E2E goldens, plus a new `TestMachine_lowerToAddressMode_scaledIndex` unit test. **Measured** by emitted arm64 code size (the host is amd64, so wall-clock isn't meaningful; code size — 4 bytes/instr — is the deterministic, host-independent metric; each fold removes an lsl+add = 8 bytes). Producer modules compiled under qemu-arm64, before→after: wasi_cargo 210440→209464 (**−976 B, −244 instr, −0.46%**), greet_rust 19672→19592 (−80 B, −0.41%), wasi_zigcc 101832→101736 (−96 B), greet_tinygo 147492→147460 (−32 B); greet_zig unchanged (its codegen doesn't emit the shifted-index address pattern). A memory's declared `min` is a permanent static lower bound (memories only grow, never shrink), so any load/store whose base is a compile-time constant and whose end (`const + constOffset + opSize`) is `<= memoryMinSizeInBytes` can never trap. `memOpSetup` now recognizes this case (via `builder.InstructionOfValue(baseAddr).Constant()`) and skips the length-load + compare + `ExitIfTrue` entirely, still recording the known-safe bound so later accesses off the same base reuse the address. Valid for shared/imported memory too — the import's `min` is a type constraint on the supplied memory. Common in interpreter-style modules whose state lives in globals at fixed addresses inside hot loops (where the per-block known-safe-bound memoization is invalidated at loop headers). Upstream measured ~40% wall-time on a TeX (tectonic) module reaching Cranelift parity, ~2x smaller hot-function code. Correctness verified: full spec suite (v1/v2 memory OOB traps, threads/shared-memory atomics) passes — elision only removes checks that can never fire; 8 atomic golden fixtures in `frontend_test.go` regenerated (bounds check gone, alignment check retained). **Measured** on a quiet i9-12900HK, interleaved benchstat n=10 (`benchmarks/vs-wazero`, new `BenchmarkConstAddrLoads` constant-address-load kernel): wazy execute **5.09µs → 1.62µs, -68.1%** (p=0.000) vs main; the unchanged upstream-wazero line moved 0% (5.10µs → 5.10µs), a clean noise-floor control. Against wazero on the same kernel wazy is **~3.1x faster** (wazero has no #2514). Bonus: caseWasm compile -3.5% (p=0.000, fewer instructions to lower). No regression elsewhere — fib execute (no memory ops), instantiate, host-call all flat. Frontend-only (`frontend.go` + `lower.go`, +37 lines). Composes with C4 const-fold/CSE, which materializes the constant addresses this pass then proves safe.
- **C22. Conditional-trap exit sequence inlined at every check site** — **RESOLVED**: ported from upstream wazero [#2515](https://github.com/tetratelabs/wazero/pull/2515). Each conditional trap (bounds check, div-by-zero, unaligned atomic, …) previously inlined the full exit sequence (set exit code + save SP + save return addr + exit) at the check site, with the guard branch jumping *over* it — so the hot path took a branch on every successful check and the ~6-instruction sequence was duplicated at every site. Now the check branches to a per-function **trap island** shared by all sites with the same exit code, and the hot path falls through (the branch is taken only when trapping). Islands are materialized once, after register allocation, by `emitTrapIslands` (in `PostRegAlloc`) using only real/dead registers on a path that never returns: **arm64** hands the exec-context over in the reserved tmp register (`x27`) and uses `x17` as scratch; **amd64** has no reserved scratch, so the site stores the exec-context to the word just below RSP (`[rsp-16]`, dead — no call/push intervenes and signal handlers use their own stack) and the island reloads it into RAX with RBP as scratch. When source-offset info is tracked (debug builds, `needSourceOffsetInfo`) the inline lowering is kept so recorded trap addresses still map to the individual site. **Measured** (quiet i9-12900HK, interleaved benchstat) code size for a 16-distinct-base-load function (16 bounds-check sites, one shared island), before→after: **amd64 1206 → 669 B (-44.5%)**, **arm64 1436 → 712 B (-50.4%)**. On the real caseWasm module: **compile -6.9%** (9.40 → 8.75 ms, p=0.007) and its generated-code allocation **-7.2% B/op** (2.189 → 2.031 Mi) — less machine code to emit and encode; the wazero control moved 0%. Execute wall-time is a wash on this large-i-cache CPU (geomean -0.23%): fib shows a stable +3–6% that is a code-**layout** artifact, not a regression — fib has no `ExitIfTrueWithCode`, so `emitTrapIslands` early-returns and its codegen is byte-identical; it merely relocated to a different cache-line alignment because the other functions shrank. The store/mov-per-site tradeoff pays off in code size + i-cache, the wall-time win growing on smaller instruction caches (upstream: -34% total code on a tectonic module, ~9% wall-time on an i7-3770). Correctness verified on **both ISAs**: full amd64 suite + arm64 under qemu (spec v1/v2 OOB traps, div-by-zero, threads, EH, tail-call, typed-func-refs; backend `TestE2E` finalize goldens regenerated for `memory_load_basic`/`memory_stores`/`AtomicRmwAdd`, incl. the `memory_load_basic` island lowering). Backend-only (`isa/{arm64,amd64}/machine.go`, `lower_instr.go`/`machine.go`, `machine_pro_epi_logue.go`). Complements C21: C21 removes provably-safe checks, C22 cheapens the ones that remain.

- **C23. `call_indirect` inline cache (per-site monomorphic target cache)** — **VALIDATED, NOT PROCEEDING (Phase 0 gate).** Proposed: split each `call_indirect` into a guarded fast path (compare the loaded `funcInstancePtr` against a per-site cached pointer; on hit, direct-call the cached executable/moduleCtx, skipping the null check + both typeID loads + type compare + exec/moduleCtx loads + converting the indirect branch to a direct one) plus the original slow path (self-populating the slot on miss). Safe under `table.set` when guarding on `funcInstancePtr` (always loads the current table entry; a mutated slot simply misses and refills). **Gated out by measurement**: the win only exists on tight monomorphic dispatch loops with near-empty callees, a regime absent from every real target workload here. Built a purpose-made dispatch kernel (`benchmarks/vs-wazero/testdata/dispatch.wat` + `dispatch_test.go`: 8 same-typed funcref "methods", loops `mono`=always slot 0, `poly`=slot `i&7`, `direct`=direct call, plus `mono_heavy`/`direct_heavy` with a ~16-op method body). Measured on the i9-12900HK, core-pinned (`taskset -c 2`), 100k-iter/op, n=4 (machine was under load — a `test262.test` at ~985% CPU — so absolute ns are noisy, but the **ratios** below are load-robust because pinned single-thread runs take load symmetrically; the deltas are the gating quantity):
  - tiny callee: `mono` **3.34 ns/call** vs `direct` **1.73** → dispatch overhead **+1.61 ns (48% of the loop)**; this is the *entire* overhead an IC competes against.
  - heavy (~16-op) callee: `mono_heavy` **10.54 ns/call** vs `direct_heavy` **10.92** → dispatch overhead **−0.38 ns, i.e. below the noise/layout floor** — a real method body already dilutes the fixed dispatch cost to nothing.
  - `mono` **3.34** vs `poly` **3.65** → the megamorphic penalty (indirect-branch misprediction, the part an IC's direct branch removes) is only **0.31 ns (8.5%)** on this BTB; most of the dispatch cost is the dependent-load chain, not the branch.

  An IC's *fast path still keeps* the bounds check (len load + compare + branch), the `table[index]` load, one pointer compare, `storeCallerModuleContext`, and a call — so it can only recover the null check + 2 typeID loads + type compare + exec/moduleCtx loads (~4 loads + 2 branches), roughly **half** of the 1.61 ns. Best-case ceiling: `mono` 3.34 → ~2.4 ns (**~25–30% on this pathological kernel only**), and **sub-noise (~0%) on `mono_heavy` and on every non-dispatch-bound real workload**. Workload reality: the vs-wazero suite (caseWasm fib/matmul/base64, constaddr, dynaddr, hostcall, spin) has no dispatch-hot loop; the real WASI guests (`cargo-wasi/wasi.wasm` has 110 `call_indirect` *sites*, `zig-cc` 14) run `_start` once, so `call_indirect` is ~0% of their runtime — a hot dispatch loop had to be synthesized to make it measurable at all. Per-site monomorphism instrumentation was therefore not run: the gate is an OR and the "`call_indirect` is <~a few % of runtime on the target workloads" arm already fires; monomorphism only matters where the sites are hot, and they are not. Against that ~25%-on-a-microbenchmark / ~0%-in-practice upside sits a real correctness surface: a per-instance mutable cache slot plumbed through moduleCtx/execCtx offsets, `table.set` staleness (miscompile if guarded wrong), a new SSA fast/slow block shape, and both-ISA golden regeneration. **Decision: do not implement.** It would become attractive only if (a) a genuinely dispatch-bound guest (e.g. a bytecode interpreter with tiny handlers in a tight `call_indirect` loop — note many such producers emit `br_table`, not `call_indirect`) enters the benchmark set, **and** (b) it is paired with an inliner (which wazy lacks) so the cached target can be inlined — the load-chain elision alone is not worth the surface. Higher-value unresolved compiler items to prefer: **C3** (arm64 scaled addressing into loads, "high, certain"), **C5** (phi/dominator worklist), **C8/C9** (arm64 const materialization + prologue pairing). The kernel + bench are kept in `benchmarks/vs-wazero` as reusable evidence and a ready harness should a real dispatch workload appear.
- **C24. Stack-overflow check emitted in every function prologue, even leaves** — **RESOLVED**: motivated by disassembly against wasmtime (Cranelift), which omits the check on leaves. wazy emitted `insertStackBoundsCheck` unconditionally (`machine_pro_epi_logue.go`, both ISAs); on a 1-load leaf that is a ~560-byte frame probe + `cmp`/branch + grow trampoline — most of the function. A **leaf** (makes no call of any kind — direct/indirect/tail/host-import — so `maxRequiredStackSizeForCalls` stays 0, verified via `prepareCall`) whose frame fits within the H7 MARGIN its caller already reserved below it cannot overflow: by the invariant in `backend/go_call.go` point 3, every callee's frame base sits ≥ MARGIN (512) above `stackBottomPtr`, and a leaf never re-establishes that margin for a callee (it has none), so eliding its check cannot compromise a deeper frame (`canSkipStackBoundsCheck`, gated on `frameSize()+128 ≤ MARGIN`). **Measured** (i9-12900HK, core-pinned, interleaved n=8, `benchmarks/vs-wazero` execute kernels): **−16% geomean** — `dynaddr` **−37%** (its leaf `$ld` is called 16 000×), `dispatch_mono/poly` **−19%/−22%**, `spin` −4%; a tiny leaf whose own hot loop gets realigned can swing (`constaddr` +8% in isolation, washed out combined with C25). Moves `dynaddr` 0.60→0.93× and `dispatch` 0.75→0.92× of wasmtime. Both ISAs verified incl. arm64/qemu (`huge_call_stack_unwind`, `TestNativeDeepRecursionWithHostCalls`, terminate-on-close, spec v1/v2). Portable, pure-Go, no golden changes (no leaf in the finalize goldens).
- **C25. Bounds check kept for range-bounded (masked) dynamic addresses** — **RESOLVED**: generalizes C21 (which elided only *constant* bases). A base address of the form `x & C` is bounded above by the constant mask `C` (only bits set in `C` can be set in the result), so if `C + constOffset + opSize ≤ memoryMin` it can never trap — the same "min is a permanent static lower bound" reasoning. `memOpSetup` now recognizes an `OpcodeBand` base with a constant operand and elides the length-load + compare + `ExitIfTrue`, covering the ubiquitous masked-index idiom (`h & (n-1)`, ring-buffer `i & mask`, `& 0x…`) that Cranelift's value-range analysis already handles. Frontend-only (`lower.go`), so **both ISAs**. Combined with C24, `dynaddr`'s `$ld` dropped **160 → 32 bytes** (≈ wasmtime's 26; identical shape: `imul; and; load-base; load` with no stack or bounds check), and the two together **flip `dynaddr` from wasmtime 1.7× ahead to wazy 1.18× ahead**, and bring `dispatch` to parity (0.94–0.97×). Correctness: elision fires only when provably in-bounds (`C+ceil ≤ min`), so it cannot hide a real OOB; verified by spec v1/v2 OOB traps + a dedicated e2e (`masked_bounds_e2e_test.go`: masked-safe load elided & correct, large-mask OOB still traps, both ISAs). Mirrors C21's 32-bit-address assumption. Future extension: other range-narrowing ops (`Uremu` → `< C`, shifts).

Impact: med-high, certain. — **RESOLVED**: `newTcpConn` now caches `tc.SyscallConn()` once into a `syscall.RawConn` field, so Read/Write/Recvfrom skip the per-call `SyscallConn` allocation; the `control` helper collapses the old two-closure structure (`syscallConnControl`'s adapter + the caller's `fn`) into a **single** closure matching `RawConn.Control`'s `func(uintptr)` signature (results via captured named returns, inner-errno-wins priority preserved). `RawConn.Control` is kept (fd stays valid for the syscall — the bare-fd caching alternative was rejected as unsafe). Falls back to `SyscallConn` per call only if construction's `SyscallConn` failed (nil cache). Measured (`BenchmarkTcpConnWrite`, nonblocking write): **5 → 2 allocs/op, 106 → 80 B** (alloc counts are load-independent; the box was busy so ns/op wasn't trusted). The floor of 2 is the one `op` closure `Control` requires plus its captured-var home. Covered by `TestTcpConnFile_Read/Write/Stat/SetNonblock` + wasi `sock_test`; full `go test ./...` green.
- **W5. Every Stat = `os.fileStat` alloc + fresh `*cachedStat` alloc** (`osfile.go:174-187`). Fix: `syscall.Fstat` into value field. Impact: medium, certain. — **PARTIAL/RESOLVED (safe half)**: `cachedStat` is now stored **by value** with a `valid bool` (both `osFile` and `fsFile`), so a Stat call no longer allocates a fresh `*cachedStat` per call. `BenchmarkOsFileStat`: **2 → 1 alloc/op, 232 → 208 B**. The residual 208 B alloc is the `os.fileStat` returned by `os.File.Stat()` inside `defaultStatFile`; eliminating it needs a raw `syscall.Fstat` + a per-platform re-implementation of os's `syscall.Stat_t → fs.FileMode` conversion (mode/symlink/device bits) — disproportionate correctness risk for one alloc, so **left as the ceiling**. Never-invalidated cache (open-lifetime fields dev/ino/isDir), so no reset sites to guard.
- **W6. `fd_fdstat_get` does a full fstat just for the immutable filetype** (`fs.go:186-234`). Fix: cache filetype on `FileEntry`. Impact: medium, certain.
- **W7. `atPath` allocates 1-3 strings per path_* call**, plus `dirFS.join` concat + `BytePtrFromString` — 3-4 copies of the same bytes. Fix: single-buffer build. Impact: medium, certain. — **INVESTIGATED, NOT DONE (bad trade).** Measured `Benchmark_pathFilestat` (os.DirFS, fd=root): 6 allocs/op, 336 B. Breakdown of the common preopen path: `string(b)` (atPath copies guest memory — **safety-required**, aliases mutable guest memory and is returned to 13 callers), `cleanedDir + path` (`dirFS.join`, needed because `os.Lstat`/`os.OpenFile` take a `string`), `BytePtrFromString` (inside the syscall, os stdlib), `*os.fileStat` (the returned FileInfo — same ceiling as W5), plus `fn.Call` machinery. **atPath itself is already 1 alloc (`string(b)`) in the common preopen+clean case** — the "1-3 strings" is the rare worst case (non-preopen subdir fd + needs `path.Clean` + trailing slash). None of the remaining copies is safely removable without abandoning the portable `os.*` API. The note's "single-buffer build" requires giving `dirFS` (currently string-only: `dir`/`cleanedDir`, **no dirfd**) an open directory fd and rewriting every op onto fd-relative `*at` syscalls with a pooled buffer — platform-specific, and must preserve wasi-testsuite `interesting_paths` semantics (symlink escapes, `..`, `O_DIRECTORY`, trailing-slash-on-file). That is a day-scale, high-risk rewrite for ~2 allocs on a path dominated by an actual filesystem syscall (~1-2µs) — deliberately deferred (wazero uses `os.*` precisely to avoid reimplementing this).
- **W8. `poll_oneoff` heap-allocates one `*event` per subscription** (`poll.go:109`, escape-verified) — this is every sleep/timer tick in Rust/TinyGo guests. Fix: write records into outBuf directly. Impact: medium, certain. — **RESOLVED**: `event` (~40 B; `userData` slices guest memory, no copy) is now a **value**, not a heap `&event{}` — `writeEvent` takes it by value and `blockingPollSubs` stores it by value, so poll_oneoff no longer allocates N `*event` per call. Measured via `testing.AllocsPerRun`: **8 clock subs = 4.0 allocs/op** (constant `fn.Call` overhead, independent of N; previously ≥ N). All poll_oneoff tests (incl. multi-sub, out-of-order, blocking-pipe) still pass.
- **W9. DirentCache first-read reallocates the window; dot entries force growth** (`sys/fs.go:184-201`). Low-med.
- **W10. `path_open` re-looks-up the fd it just created** for the O_DIRECTORY check. Low.
- **W11. `fd_prestat_dir_name` copies via throwaway `[]byte`** — **RESOLVED**: `WriteString`.
- **W12. `random_get` = one getrandom(2) per guest call** with real entropy configured; batch/buffer. Low-med.

## Component runtime (`internal/component`)

The WASI 0.2 component layer sits above the core engine; its hot paths are
Instantiate (serverless: a fresh instance per request) and per-call marshalling
(long-lived instance). Wins here are measured by **allocs/op** (deterministic,
load-independent) on `internal/component/instance` `BenchmarkInstantiate*` /
`BenchmarkCall*`.

- **CM1. CompileCache only cached compiled core modules — the component binary was re-parsed every Instantiate** — **RESOLVED.** `binary.Decode` ran on every `Instantiate`, and `finalizeBoundExport` re-flattened/re-resolved every export's ABI, even though both are pure functions of the immutable component. CompileCache now also caches (a) the decoded `*binary.Component` keyed by the component bytes (`getOrDecode`; hit path uses the `map[string(byteslice)]` no-copy idiom so a hit allocates nothing, crucial for tens-of-KB components), and (b) per-export ABI metadata (`*boundExportABI`: flatten + type-resolution) keyed by the shared comp pointer + func index (`abiFor`). Both are immutable and shared read-only across all Instances (and concurrent instantiations — `-race` clean). Split `finalizeBoundExport` into the pure `computeBoundExportABI` (cacheable) + per-instance handle resolution. **Measured** (allocs/op, deterministic): `InstantiateCached` **199 → 79 (−60%)**; `InstantiateHelloCached` 3588 → 3037, −66 KB/op. The residual instantiate allocs are inherent per-instance engine work (`Store.instantiate`, `FunctionInstanceReference`, `NewFunction`, `buildGlobals`, `moduleConfig.clone`, per-instance export index) — each instance genuinely needs its own module instance/globals/function refs. Gated on a `CompileCache` (no cache ⇒ decode + compute fresh, unchanged). Oracle + 35/35 conformance green.
- **CM9. Per-canon host modules + guest→host call path (tier-2/3 of the two-model audit)** — **RESOLVED (host-module merge + import-path compile); two sub-items deferred with reasons.** (a) *Guest→host WASI call path compiled.* The export direction had bind-time ABI plans (CM3/CM6), but the import direction — the highest-frequency path (every WASI stream write, clock, fd op) — re-derived everything per call: `liftHostArgs` did `resolveTypeRef` + `abi.Flatten` (fresh `[]string`) + `usesMemory` walk + borrow detection per param, `lowerHostResults` re-resolved the result type per call. `buildHostWrapper` now precomputes these once (`buildHostParamPlans`/`buildHostResultPlan`) and the closure uses `liftHostArgsPlanned`/`lowerHostResultsPlanned`. **Measured**: `BenchmarkLiftHostArgs` (string+u32) **8 → 5 allocs/op (−38%), 250 → 122 B (−51%)**. (Full `LiftStep`/`LowerStep` reuse is blocked by the param-vs-result spill threshold difference — host args arrive flat under MaxFlatParams=16, CompileLift bakes in MaxFlatResults=1 — so this hoists the bind-time work and keeps the oracle-verified `LiftFlat`/`LowerFlat` for data movement.) (b) *Per-canon host modules merged.* Each core-func-producing canon (lower / resource.{new,drop,rep}) was its own `NewHostModuleBuilder` + compile + Instantiate + `sys.Context` (~19 for real_hello, ~31% of cached-instantiate alloc objects). `buildCanonHostModule` split into `computeCanonHostFunc` + `buildMergedCanonHostModule`; the inline-export loop packs a whole group's canon funcs into one module (`f<i>` exports). **Measured**: `InstantiateHelloCached` **2606 → 1995 allocs/op**. **Deferred:** *stable-name shim caching* (make canon shim refs component-constant so the ~13 passthrough shims hit the compile cache instead of recompiling every instantiate) — attempted, reverted: the per-instance `ImportResolver` does `v.(*wasm.ModuleInstance)`, but host modules are the `hostModuleInstance` wrapper, so resolver-based host-module resolution panics; making it work needs a wazy-package change to unwrap host modules, touching the ExportedFunction-forbidden boundary, for a warm-cache-only benefit — not worth the risk. *Component memory pooling* (component memories set `imported`, bypassing the buffer pool) — deferred: needs refcounting in shared core-runtime `module_instance.go` (use-after-free class) for a timing benefit the audit flagged as unproven (fresh 1 MB `make` gets OS-zeroed pages, a pool hit pays an explicit clear).
- **CM8. getOrCompile copied the whole core module on every cache HIT** — **RESOLVED.** `getOrCompile` materialized `key := string(coreBytes)` before the lookup, copying up to ~250 KB/op on the real_hello graph every hit; now looks up via `c.byKey[string(coreBytes)]` (compiler elides the copy for a map index) and materializes the key only on the miss/store path, mirroring `getOrDecode`.
- **CM7. Component import-discovery double-compiled every core module** — **RESOLVED (this is CM5's tractable lever).** `discoverNeededFuncTypes` ran a full native `CompileModule` on every embedded core module just to read its import signatures, then threw it away — so each module was compiled TWICE per Instantiate (the CPU profile showed the probe pass at ~43% cumulative). An import's module/field/type is in the *decoded* structure, so discovery now decodes only (`runtime.DecodeModuleNoCompile`, reading `TypeSection` by the import's own type index with a bounds check that also sidesteps the post-validation-only `buildFunctionDefinitions`); the real compile happens once. Discovery + the empty-import rewrite are also cached per component (`CompileCache.graphPlanFor`), and the rewritten core bytes are reused by the main loop (drops the double-rewrite even with no cache). `paramHasResource`/`paramsSpill`/`paramTuple` moved into the cached `boundExportABI`. **Measured**: `InstantiateHello` **~28 ms → ~14 ms** cold (quiet box), **22283 → 13111 allocs/op (−41%)**; `InstantiateHelloCached` 3331 → 2606 allocs/op with the caching. Race + 7/7 conformance + vet + staticcheck green; coverage 90.3%.
- **CM6. Canonical ABI lift tree-walked the type descriptor on every call (result side)** — **RESOLVED.** The symmetric counterpart of CM3: `liftResult`'s flat path called `LiftFlat`, which re-`Flatten`ed the result descriptor per call (to re-derive the spill decision) and drove a `CoreValueIter` type-switch even to lift a single scalar. All invariant for a fixed result type. `abi.CompileLift` now compiles the result into a `LiftStep` at bind time (cached on `boundExport.resultStep`): a scalar primitive lifts from its one core value directly (no iterator, no `Flatten`); own/borrow is a bare i32 handle; a result flattening past `MaxFlatResults` (string/list/multi-field aggregate) is `Load`ed from the returned pointer; only a genuinely-flat aggregate keeps the tree-walk body, minus the redundant `Flatten`. `liftResult` dispatches through `be.resultStep.Lift(coreResults, mem)`. Verified equal to `LiftFlat` over the full flat oracle battery (`TestLiftStepMatchesLiftFlat`) — transitively spec-conformant via definitions.py. Combined with CM3 this makes the ABI marshalling compiled-to-direct-ops in **both** directions; the leaf shapes real WASI signatures use (primitive/string/handle) carry no per-call type-switch or `Flatten` on either param or result. Remaining Go interpretation is deferred with reasons: composite-by-value marshalling (rare in WASI; a recursive plan is marginal payoff), `resolveArgHandles`' resource tree-walk (gated on resource params only), and the primitive-kind string switch (a 2–6-char switch Go already compiles tight; int-jump-table rewrite is ~1–2% of a scalar call for a 13-case×2-direction refactor — not worth it).
- **CM3. Canonical ABI lower tree-walked the type descriptor on every call** — **RESOLVED (param side).** `LowerFlatInto` per-call type-switched on the descriptor, called `Flatten` (a `[]string` alloc) to decide spilling, and returned intermediate `[]CoreValue` slices from each recursion level — all invariant for a fixed param type. `abi.CompileLower` now compiles each param into a `LowerStep` at bind time (cached in `boundExportABI`, shared read-only across instances): primitive/string/handle leaves (the WASI-common shapes) execute a direct op into the destination buffer (no type-switch, no `Flatten`, no intermediate slice); composites keep the tree-walk with the spill decision precomputed. `lowerParams` dispatches through `be.paramSteps[i].Lower`. Verified byte-for-byte equivalent to `LowerFlatInto` over the full flat oracle battery (CoreValues + linear memory, same bump allocator) — transitively spec-conformant via the definitions.py oracle. **Measured**: CallGreet **8 → 7 allocs/op, 190 → 141 B/op** (per-call Flatten `[]string` + intermediate `[]CoreValue` gone; param lower now flat-alloc-free bar CM2's realloc closure); CallAdd unchanged at 2. The lift/result side is now specialized too — see CM6. A `-> string` result's two allocs (byte-copy out of guest memory — mandatory since it may be freed post-call — + `Value=any` boxing) are inherent. A full *recursive* composite plan (vs. the current precomputed-spill + tree-walk body) is a possible future extension with marginal real-world payoff.
- **CM2. Per-call `abi.Realloc` closure escapes on string/list params** (`instance.go` `reallocOfFunc`) — **RESOLVED (as a struct; the "escapes" premise was stale).** `abi.Realloc` is now a struct carrying `Ctx` + a bind-time-cached `Call` func (built once on the boundExport), so per-invoke it is a stack struct literal instead of a closure; a `ReallocFunc` adapter wraps simple/test allocators. Escape analysis (`-gcflags=-m`) shows the struct does NOT escape — and, re-measured, neither did the *old* closure (Go's escape analysis had improved since this item was filed), so the change is **alloc-neutral**. Kept anyway: it removes per-call closure *construction*, builds the realloc callback once, and carries `ctx` cleanly without the wide signature sweep the original DEFERRED note feared. The realloc call itself is now alloc-free (see CM4).
- **CM5. Cached instantiate re-decodes + re-compiles per instance** (`InstantiateHelloCached` ~3000 allocs / 2.2 MB even with the guest compile cached) — **RESOLVED — see CM7 (re-decode cached + double-compile removed) and CM9 (host-module rebuild reduced by merging). Original profiling notes retained below.** Top alloc sites on the cached-instantiate path: `wasm/binary.decodeImportSection`/`decodeExportSection`/`readCoreWasmName` (~20% — the core module's import/export sections are re-decoded every instantiate, for import discovery + the empty-import rewrite), `native.engine.compileHostModule` (~14% cum — the WASI host stub modules are rebuilt+recompiled per instance), and per-core-instance `NewModuleConfig`/`NewFakeRandSource`. Instantiate is one-time per instance (amortized when reused), so this ranks below the per-call path — but a per-request-instance workload would want it. The host-module recompile is hard to cache (per-instance closures capture WASIConfig/resources); the re-decode is the more tractable lever (cache the graph engine's per-component decoded import/export analysis alongside the compiled module in the CompileCache). Deferred until instantiate latency is a stated goal.
- **CM4. Component call path used `api.Function.Call` (allocates a `[]uint64` result slice per invocation)** — **RESOLVED.** `invoke` now uses `CallWithStack` for all three engine calls on the export path — the guest core func (pooled param buffer sized to `max(params, results)`, results read back in place), post-return (reuses `rawResults`' own buffer), and `cabi_realloc` (a fixed `[4]uint64` stack buffer) — removing the result-slice allocation `Call` makes each time (exactly the alloc C10 says CallWithStack exists to avoid, applied at the component call site). The invalid-component fail-loud branch is preserved by slicing the buffer to the core func's ACTUAL declared result count (`boundExport.coreResultCount`, resolved once at bind time), so `liftResult`'s length checks still fire. **Measured** (deterministic alloc counts, load-independent): **CallAdd 2 → 1 allocs/op, 42 → 17 B**; **CallGreet 7 → 4 allocs/op, 145 → 84 B**. CallGreet's residual 4 (result string byte-copy, `Value=any` boxing, param string lower, caller `[]abi.Value`) are semantically required. Guards: full component suite `-race`, vet, staticcheck, 7/7 conformance. (Timing improved too but the box was loaded ~13; alloc counts are the reliable metric.)
